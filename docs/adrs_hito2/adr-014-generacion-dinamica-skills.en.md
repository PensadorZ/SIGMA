# ADR-014: Dynamic Generation of New Skills on Demand

## Executive summary of v1.2 changes

**Reconciliation with the real hierarchy (ADR-016).** v1.1 spoke of a
generic "Orchestrator (Architect mode)" as the author — terminology
predating the Director/Engineer/Auditor hierarchy. Roles are
reassigned: **Engineer Auditor detects** the need (via
`0013-skill-discovery`), **the Director authorizes** (sole point of
contact with global HITL, ADR-016 Tab. 1), **the corresponding domain
Engineer builds** — never an abstract "Orchestrator." A link to ADR-017
(sandboxing) is added: every dynamically generated skill runs inside an
ephemeral container until its promotion to production.

## Executive summary of v1.1 changes

The Context section is expanded to first explain that this ADR is the
ecosystem's highest-risk scenario — the only one where the system
writes its own code — and that it introduces no new mechanisms, but
rather orchestrates ADR-003, ADR-004, ADR-005, ADR-009, and ADR-012's
together at their point of greatest demand.

## Executive summary

This ADR defines the process by which the SIGMA ecosystem can
autonomously generate new skills when the existing catalog does not
cover a detected functional need. **Engineer Auditor detects the
gap** (via `0013-skill-discovery`); **the Director authorizes** the
generation; **the corresponding domain Engineer** (Data, Models, or
Auditor itself if the gap is in auditing) **generates the new skill**.
The new skill goes through the standard validation cycle of the Policy
Server, Green Team, Approval Endpoint, sandboxing (ADR-017), and
versioning per ADR-012. This ADR applies across all cost variants, with
Dev submode disabled (see Tab. 2).

## Context

Dynamic Skill Generation is the highest-risk scenario governed across
the whole ecosystem: it is the only point where the system itself
writes new code that will later execute with real authority. That is
why this ADR introduces no new mechanisms — it orchestrates, at its
point of greatest demand, all those that already exist: ADR-009's
specification, the Policy Server's validation (ADR-005), the Green
Team's tests (ADR-003), ADR-004's approval, ADR-017's sandboxing, and
ADR-012's versioning. If any of those mechanisms had a flaw, this is
where that flaw would have the greatest possible impact.

SIGMA's skill catalog is initially finite. In an ecosystem operating
across domains as diverse as data science, engineering, physics,
mathematics, or philosophy, uncovered functional needs will inevitably
arise. Without a formal mechanism, the system remains limited to its
initial catalog, losing its character as an autonomous, adaptable
ecosystem.

**Why the v1.2 role reassignment is not cosmetic:** with a generic
"Orchestrator" as author, it is unclear *which Engineer* takes
responsibility for the new code once promoted — a Data Science skill
generated by an abstract actor has no clear owner within ADR-016.
Assigning authorship to the corresponding domain Engineer solves that:
the generated skill integrates into the Engineer that will actually
operate it, with the same responsibility as its manual skills.

## Decision

### Fig. 1 — Flow for dynamically generating a new skill

```
STEP 1 — Detecting the need
─────────────────────────────────────
Detection sources:
  A. A user query no skill can resolve
     └─ The Director receives it (intent interpreter, ADR-016 Fig. 1)
        and logs it in Langfuse as a coverage failure
  B. A recurring failure pattern detected in Langfuse
     └─ Engineer Auditor, via 0013-skill-discovery, detects it
        systematically — does not require the Director to notice first
  C. An explicit operator request via the Approval Endpoint
     └─ Goes directly to the Director
        │
        ▼
Director consolidates the creation mandate and assigns the
corresponding domain Engineer:
  {functional description, domain constraints,
   estimated impact level, ADRs governing the new skill,
   assigned Engineer (Data | Models | Auditor)}

STEP 2 — Generation by the assigned domain Engineer
─────────────────────────────────────
The assigned Engineer generates, inside an ephemeral container
(ADR-017 — never outside the sandbox, with no exception):
  ├─ Full SKILL.md (ADR-009 format)
  │    ├─ YAML frontmatter (version: gia_0.1.0)
  │    ├─ Gherkin scenarios (≥1 positive, ≥1 negative)
  │    ├─ LTL properties (≥1 safety, ≥1 liveness)
  │    └─ Traceability specification
  └─ skill.py (initial implementation)
Temporary storage: skills/generated/{skill_id}/

STEP 3 — Review by Engineer Auditor
─────────────────────────────────────
Engineer Auditor (via 0012-code-reviewer) verifies:
  ├─ Coherence of expected_trajectory with existing skills
  ├─ Formal correctness of LTL properties
  └─ OutputSchema does not violate K⊆X (ADR-008)

STEP 4 — Validation by the Policy Server (ADR-005)
─────────────────────────────────────
  ├─ Structural layer: authorized tools, permitted dependencies
  └─ Semantic layer: PII, credentials, intent deviation

STEP 5 — Testing by the Green Team (ADR-003), inside the sandbox
─────────────────────────────────────
  ├─ Runs the skill in ADR-017's ephemeral container
  ├─ Verifies Gherkin scenarios
  ├─ Verifies LTL properties at runtime
  └─ Static analysis with the code-reviewer skill

STEP 6 — Approval (ADR-004)
─────────────────────────────────────
  ├─ LOW impact  → Vibe Diff generated but no approval required
  └─ MEDIUM/HIGH impact → Vibe Diff + operator approval (Director
     escalates to global HITL — sole point of contact, ADR-016 Tab. 1)

STEP 7 — Versioning and promotion (ADR-012)
─────────────────────────────────────
  Initial version: gia_0.1.0
  Cycle: Dev → Staging → Production
  (same criteria as any skill)
  The skill leaves ADR-017's sandbox only upon reaching Production,
  with a green Green Team history
```

### Tab. 1 — Roles and responsibilities in dynamic generation

| Agent | Role | Responsibility |
|---|---|---|
| **Director** | Authorizer | Consolidates the mandate, decides whether to authorize (global HITL if impact requires it), assigns the domain Engineer |
| **Assigned domain Engineer** (Data / Models / Auditor) | Author | Detects via source A (if the Director is who notices) or receives the mandate, drafts `SKILL.md` and `skill.py` inside the sandbox |
| **Engineer Auditor** (`0013-skill-discovery`) | Systematic detector | Detects recurring failure patterns in Langfuse (source B) |
| **Engineer Auditor** (`0012-code-reviewer`) | Reviewer | Verifies the generated skill's trajectory coherence, LTL, and K⊆X |
| **Policy Server** | Structural/semantic validator | Validates tools and dependencies in the generated skill |
| **Green Team** | Functional validator | Runs Gherkin/LTL tests inside ADR-017's sandbox, reviews code with `code-reviewer`, manages the Vibe Diff |
| **Blue Team** | Registrar | Updates the AgBOM when the skill is promoted to production |

**Note:** Engineer Auditor appears twice with distinct
responsibilities (systematic detection via `0013`, review via
`0012`) — these are different skills within the same Engineer, not a
role conflict.

### Tab. 2 — Behavior by cost variant

| Variant | Status | Approval level |
|---|---|---|
| **SIGMA-FE** | Active | LOW impact with no human approval; MEDIUM/HIGH with approval |
| **SIGMA-LE** | Active | Same levels as FE |
| **SIGMA-ME** | Active | Same levels as FE |
| **SIGMA-HE** | Active | Same levels as FE |

**Cross-cutting submodes** (apply to any of the four cost variants):

| Submode | Status | Approval level |
|---|---|---|
| **Dev** | **Disabled** | Not applicable; prevents unvalidated skills from proliferating during debugging |
| **Runtime** | Active | **Any generation** requires operator approval, regardless of level |

## Positive consequences

- The system can adapt to new needs with no human intervention for
  low-impact cases.
- The validation cycle guarantees generated skills meet the same
  standards as manually designed ones.
- The role separation between the author Engineer and Engineer Auditor
  as reviewer follows ADR-003's separation-of-concerns principle — and
  now also assigns a clear owner (the domain Engineer) to the skill
  once promoted.
- The `gia_` version mark (ADR-009) permanently identifies a skill's
  origin for security audits.
- Mandatory sandboxing (ADR-017) contains the ecosystem's highest-risk
  scenario from the very first second of execution.

## Negative consequences

- The process adds computational and time overhead before a new skill
  is available in production.
- The generated skill's quality depends on the assigned Engineer's
  ability to correctly interpret the mandate.
- The `skills/generated/` directory requires cleanup policies for
  obsolete, unpromoted skills.

## Alternatives considered

| Alternative | Why it was rejected |
|---|---|
| Manual generation by a developer | Introduces latency and dependency on human resources |
| Generation by the Auditor | Mixes governance responsibilities with code generation, violating ADR-003 |
| A generic "Orchestrator" as sole author (v1.1) | No clear owner after promotion — doesn't fit ADR-016's real hierarchy |
| Validation by the Policy Server only | Doesn't verify functionality or code quality |
| No research repository | Prevents traceability, versioning, and rollback |
