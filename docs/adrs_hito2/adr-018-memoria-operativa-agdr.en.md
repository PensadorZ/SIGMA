# ADR-018: Cross-Run Operational Memory (Ag-DR) and Formalization of the Director's Decision Logic

## Executive summary

This ADR originates from a gap identified when comparing SIGMA against
an external agent-management framework (Agents.md/Context.md/Skills.md/
Memory.md): SIGMA has no mechanism by which the system accumulates
operational experience from one run to the next. This document defines
the **Ag-DR** (Agent Decision Record) — a record each orchestrator
generates about its own real decisions, distinct from an ADR (which
documents architecture, not execution). At Marx's explicit request, it
also formalizes that the Director's decision logic be specified with
the same rigor ADR-009 already requires of skills: schemas + Gherkin.

---

## Context

`ADR-001` (epistemic memory) separates verified facts from hypotheses
**within a single run** — it accumulates nothing across runs. No SIGMA
component today "remembers" anything from prior executions; every
Director run is a blank slate. This is a valid governance decision
within a single run, but it left the question of operational growth
completely unresolved: how could SIGMA, for instance, notice that a
certain kind of dataset triggers `PAUSED_HITL` in `0004` with unusual
frequency, without a human having to manually review Langfuse every
time.

The central risk this ADR must resolve, not ignore: any memory
mechanism an LLM drafts freely (the Director, via Gemini/Ollama) is a
new surface for violating K⊆X (`ADR-008`) — a poorly designed Ag-DR
could become plausible but unverifiable narrative, exactly what K⊆X
exists to prohibit.

---

## Decision

### 2.1 — Definition of Ag-DR

An **Ag-DR** (Agent Decision Record) is a structured record an
orchestrator (Director, Engineer Data, Engineer Models, Engineer
Auditor) generates at the end of a run, documenting **what decisions it
made and on what evidence**, not system architecture. It is to the
Director what an ADR is to the project: an ADR says "why we chose
LangGraph"; an Ag-DR says "in run `sigma-20260713-a1b2c3d4`, Engineer
Data selected the `descriptive_fallback` branch in `0004` because there
was neither `hypothesis` nor `datetime_index` in the state."

**Non-negotiable rule (link to ADR-008):** an Ag-DR is built
**exclusively** from the already-structured fields each skill emits via
`emit_trace_event()` (`branch_selected`, `verdict`, `duration_ms`,
`error_type`, etc.) — **never** from free text generated by the
Orchestrator's LLM. The LLM may, at most, *format for readability* data
that is already verified — never *invent* the record's content. This is
the operational translation of K⊆X to this new artifact.

### 2.2 — Folder structure

```
sigma/
└── memory/
    ├── director/
    │   └── {trace_id}.agdr.md
    ├── engineer_datos/
    │   └── {trace_id}.agdr.md
    ├── engineer_modelos/       ← empty until Rollout 2
    │   └── {trace_id}.agdr.md
    └── engineer_auditor/       ← empty until Rollout 3
        └── {trace_id}.agdr.md
```

One folder per Engineer, as Marx proposed — each Engineer writes only
to its own, never to another's (the same isolation rule already
governing ADR-016 §2.3, "an Engineer never invokes another Engineer's
skills").

**Engineer Data generates the most detailed Ag-DR of the four** — it
handles the input data, and therefore has the most context to record
(dataset schema, `0004`'s result, whether there was an HITL pause and
why).

### 2.3 — Format: YAML (frontmatter) + Markdown

The same pattern already established in `SKILL.md` and in the ADRs
themselves — structured metadata, parseable by code, followed by a
human-readable section and, especially, **readable by the Director
itself** (which needs to be able to read its own past Ag-DRs without
having to parse raw Langfuse):

```markdown
---
trace_id: sigma-20260713-a1b2c3d4
engineer: engineer_datos
pipeline_run_id: run-20260713-a1b2c3d4
sigma_variant: SIGMA-FE
sigma_submode: Dev
skills_ejecutados: ["0000","0001","0002","0003","0004","0008","0011"]
verdicts:
  "0004": {branch: descriptive_fallback, verdict: INSUFFICIENT_EVIDENCE}
  "0008": {pct_unclear: 12.4}
pipeline_status: success
hitl_disparado: false
operador_id: marx-001   # open field, see §2.4 — not hardcoded to a single value
duration_total_ms: 48210
---

## Run summary

Engineer Data processed the dataset in `data/tirendaz.csv` without
incident. `0004-statistical-validator` fell into `descriptive_fallback`
(no hypothesis, temporal index, or live feedback available) and found
no nulls or duplicates above threshold. `0008` classified with 12.4% of
rows `UNCLEAR`, below the HITL pause threshold (30%).
```

The "Run summary" section **may indeed be drafted by an LLM** — but
only as a paraphrase of the already-structured frontmatter preceding
it, never adding information not present there. This is mechanically
verifiable (D3 of `ADR-007`, or a new adherence check): no proper noun
in the prose that doesn't also appear in the YAML.

### 2.4 — Authorship, including the human operator

Every Ag-DR records which agent generated it and under which human
operator's authorization (`operador_id` field). **Correction after
review with Marx:** this field is open by design, not a single
hardcoded value — SIGMA is an open-source project (GitHub, LinkedIn)
and anyone who clones the repository runs their own instance as that
instance's sole operator. `operador_id` does not reopen the
single-instance multi-operator architecture (which remains rejected,
`AGENTS_CREATOR.md` §1) — it simply avoids hardcoding a value that
would only make sense for Marx's own instance.

### 2.5 — Human approval as a gate before an Ag-DR becomes trusted memory

**Resolves question 2 of the original Section 6.** Same pattern as Vibe
Diff + HITL (`ADR-004`): an Ag-DR is **generated automatically** after
every run, but remains in `pendiente_revision` (pending review) state
until the operator explicitly audits and approves it. Only an Ag-DR in
`aprobado` (approved) state can, in the future, be consulted by the
Director to inform a decision — never one still `pendiente_revision`.
This prevents the system from "learning" from a defective run without a
human having validated it first.

```yaml
estado_revision: pendiente_revision   # pendiente_revision | aprobado | rechazado
revisado_por: null                    # operador_id once reviewed
fecha_revision: null
```

### 2.6 — The Director's decision logic, formalized

At Marx's explicit request: the Director's routing logic (today
implicit in `edge_after_engineer_datos` and similar functions in
`director.py`) is specified with the same rigor ADR-009 requires of
skills — **schemas + Gherkin**, not just Python code:

```gherkin
# language: en
Feature: Director routing between Engineers

  Scenario: Rollout 1 — only Engineer Data exists
    Given the Director receives a user intent
    And only Engineer Data is registered in Rollout 1
    When the Director decides which Engineer to delegate to
    Then it delegates to Engineer Data
    And it considers no other Engineer (ADR-016 §2.4)

  Scenario: Engineer Data paused on HITL
    Given Engineer Data returns "__interrupt__"
    When the Director receives that result
    Then it propagates the interrupt without generating an Ag-DR yet
    And the Ag-DR is generated only upon resuming with a definitive decision
```

Plus the corresponding LTL properties (Safety/Liveness), the same
pattern as every `SKILL.md`. This `.feature` file would live at
`sigma/core/tests/test_director_routing.feature` — pending construction
once this ADR's approved version is drafted.

### 2.7 — Worker Agent Cards embedded on decommissioning

New link to ADR-019 §2.1ter: when a Worker (an ephemeral sub-agent for a
specific task, distinct from the permanent Engineers) reaches the end
of its useful life, its full Agent Card is embedded inside the Ag-DR of
whoever created it — normally the Director. This is the only thing that
allows auditing afterward *how and why* that specific Worker was
conceived; a Worker with no such trail should not have been able to
exist.

```yaml
workers_creados:
  - agent_card: {agent_id: worker-calculador-financiero-a1b2c3, ...}
    razon_de_creacion: "One-off task with no existing skill covering it"
```

Optional field in the Ag-DR's frontmatter — empty in the vast majority
of Rollout 1 runs, since the real Worker-creation mechanism is not
implemented yet (see the scope note in ADR-019).

### 2.8 — Enrichment via automated judgment (Rollout 3), never a substitute for deterministic HITL

New link to the automated-judgment mechanism Engineer Auditor will
build in Rollout 3 (an internal design candidate for `0012`/`0015`,
with no ADR of its own yet — referenced here as an Ag-DR consumer, not
as an already-made decision about its final shape).

**Non-negotiable rule:** an automated-judgment verdict may **enrich** an
Ag-DR — attached as an additional frontmatter field, under the same
K⊆X restriction that already governs the rest of this document (2.1):
the judge answers with a claim, whether it is supported by the context,
and the exact textual citation that proves it — never free narrative
about why. An automated-judgment verdict **can never suppress** a
deterministic HITL trigger already defined elsewhere in the system
(`0008`'s threshold, `0004`'s, or any future equivalent in Engineer
Models). This preserves the same fail-safe principle already governing
`skill_runner.py`'s circuit breaker: an enrichment mechanism cannot
degrade an existing protection, it can only add information so the
operator reviews faster.

```yaml
juicio_automatico:                  # optional, empty until Rollout 3
  veredicto: null                   # aprobado | rechazado | sin_evaluar
  afirmaciones_verificadas:
    - afirmacion: null
      respaldado: null              # true | false
      cita: null                    # exact textual excerpt from the source
  suprime_hitl: false               # ALWAYS false — read-only field,
                                     # never writable by the judge itself
```

This field does not change the `estado_revision` of 2.5 — an Ag-DR with
`juicio_automatico.veredicto: aprobado` is still born in
`pendiente_revision` like any other, and still needs your explicit
human approval before it can inform a future decision. The judge speeds
up your review, it does not replace it.

## Consequences

### Benefits
- SIGMA stops "forgetting" every run — opening the door to detecting
  patterns (e.g., "`0004` drops into `PAUSED_HITL` with unusual
  frequency for datasets of a certain size") without manually reviewing
  Langfuse.
- The Director's decision logic becomes auditable the same way any
  skill is, closing a real inconsistency: today skills have mandatory
  Gherkin and the Director does not.

### Risks and mitigations
| Risk | Mitigation |
|---|---|
| The Ag-DR becomes a hallucination surface | Rule 2.1: built exclusively from already-structured Langfuse events |
| Unbounded accumulation of `.agdr.md` files | A retention policy — out of scope for this draft, still to be defined |
| The Director starts "deciding differently" based on past Ag-DRs without that being audited | Resolved by §2.5 — only an Ag-DR in `aprobado` state (reviewed by the operator) can eventually inform a future decision; one still `pendiente_revision` never influences anything |

---

## Alternatives considered

| Alternative | Why it's discarded (for now) |
|---|---|
| Letting the Director's LLM draft the Ag-DR freely | Directly violates K⊆X — it's unverifiable narrative |
| A single global Ag-DR, not one per Engineer | Breaks the isolation already established in ADR-016 §2.3 |
| Storing memory only in Langfuse, with no dedicated file | Langfuse isn't naturally suited for human/Director reading — this is what motivated this ADR |

---

## Relationship to other ADRs

| ADR | Relationship |
|---|---|
| ADR-001 | Epistemic memory is *intra*-run; the Ag-DR is memory *between* runs — complementary, not redundant |
| ADR-008 | K⊆X governs Rule 2.1 without exception |
| ADR-009 | Skills' Gherkin+LTL pattern is extended here to the Director's logic |
| ADR-011 | The Ag-DR is built from the same events Langfuse already receives |
| ADR-016 | The per-Engineer folder structure respects the hierarchy's isolation |
| ADR-019 | Each orchestrator's "Identity" format (based on CLOE) is a deliberately separate sibling ADR — opposite lifecycles: the Ag-DR is generated only per run, Identity is drafted once by a human and rarely changes |
